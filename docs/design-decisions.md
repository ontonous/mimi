# Mimi v1.0 深度设计决策与评估报告


## 一、语法兼容性规则：`.mimi` 如何与 `.mms` 共存

`.mimi` 是 `.mms` 的超集，所有合法 `.mms` 文件改扩展名后必须能直接通过 `mimi check`，但不要求能编译（即不要求所有占位已补全）。核心兼容规则如下：

| 构造 | `.mms` 中的形式 | `.mimi` 中的接受形式 | 说明 |
|------|----------------|-------------------|------|
| 类型定义 | `type A: B \| C` | 同左；另外支持 `type A: B \| C`，也支持花括号记录 | `.mimi` 允许用缩进记录体 `type A: field: T`，同 `.mms` |
| 函数体 | 必须为缩进体（`steps:` 等） | **L4 实现必须使用花括号体 `{...}`**；同时允许在函数签名后、花括号体前保留缩进的 `requires:`/`ensures:`/`math:`/`desc:` 契约前缀 | 缩进体仅用于 L1~L3 草图/步骤流；同一函数主体不能同时出现缩进实现体与花括号实现体 |
| `module` 体 | 必须为缩进体 | **L4 模块必须使用花括号体 `{...}`**；为兼容 `.mms` 草图，也接受缩进体 | `module Shop { ... }` 表示该模块已进入 L4 精确实现阶段，缩进体仅用于 L1~L3 兼容 |
| `rule` | 前置附着于实体 | 行为完全一致 | 语义不变 |
| `desc` | 独立实体 | 独立实体，也允许在花括号体内作为语句（无运行时代码） | 统一当作元数据 |
| `use` | 文件级指令 | `use a::b::c;` | Mimi 使用 `use`；`@import` 属于 MimiSpec |

**结论**：`.mimi` 完全保留 `.mms` 的缩进语法（L1~L3 草图/契约）。一旦进入 L4 精确实现阶段，**函数和模块必须使用花括号体 `{...}`**；缩进部分仅作为 L3 契约前缀或 `.mms` 兼容形式存在。这确保了从草图到生产的语义边界清晰、无破坏。

## 二、消除语法歧义：类型别名 vs Newtype，以及能力组合

### 2.1 `type A = B` 是透明别名
在 `.mimi` 中：
- `type Meter = f64;` 将 `Meter` 定义为 `f64` 的**透明别名**，两者完全互换，无任何包装。
- 需要强类型包装（Newtype）时，使用 **`newtype` 关键字**：
  ```mimi
  newtype UserId = u64;
  newtype Meter = f64;
  ```
- `.mms` 中没有 Newtype 的概念，未来如需在草图中表达，可增加 `newtype`（向后兼容）。
- `type A: B | C` 仍为枚举，`type A: field: T` 仍为记录，与原有完全一致。

**理由**：  
`type =` 作为别名已被众多语言采用；`newtype` 明确表达语义隔离，避免 `type` 多重语义导致的解析困难。`newtype` 关键字简短易学，门槛极低。

### 2.2 能力组合符改为 `+`，消除与借用 `&` 的冲突
线性能力 `cap` 的类型组合使用 **`+`** 而不是 `&`：
```mimi
cap FullFileAccess = FileReadCap + FileWriteCap;
```
分解时同样使用模式匹配：
```mimi
let (r, w) = full_cap.split();
```
- 类型上下文中的 `&T` 永远表示借用，`+` 仅用于 `cap` 的声明/组合。
- 这完全避免了解析歧义，且符合资源代数中“和”的直觉（加性组合）。

**理由**：  
借用是 Mimi 的核心概念，不能混淆。`+` 也暗示能力的“累加”，学习成本可忽略。

## 三、内存模型决策树：何时用 Move / ARC / Arena / cap

Mimi 为开发者提供分层内存策略，遵循“默认最简单，复杂按需取”原则。编译器会在适当位置提示建议。

```
需要值的所有权吗？
├─ 唯一拥有者，无需共享
│   └─ 值小且 Copy? → 自动 Copy
│   └─ 非 Copy → Move + 借用 (& / &mut)
│       注：生命周期由编译器自动推导，大部分无需显式标注
├─ 需要多个拥有者
│   ├─ 跨线程共享 → shared T (原子 ARC)
│   │   └─ 可能循环引用 → 搭配 weak T
│   └─ 仅单线程 → local_shared T (Rc)
├─ 大量临时对象，生命周期限定在作用域内
│   └─ arena { ... } 内使用 ref T（严禁逃逸）
└─ 需要静态权限控制（如文件读写）
    └─ 将 cap 作为函数参数线性传递，用后 drop
```

**对比 Rust**：  
Rust 需要从 Box、Rc、Arc、生命周期标注中手动选择，并理解 Cell/RefCell 内部可变性。Mimi 用 `shared`/`local_shared` 统一了多所有权，并内部封装了必要锁（如 `Mutex`），开发者只需在需要可变字段时声明 `mut`。Arena 提供了区域内存的一等语法，避免手动管理生命周期。总体**心智负担更低**。

## 四、深度评估与语言对比：Mimi 的差异化定位

### 4.1 意图编码到标识符（人类意愿的显式化）
`$` / `$$` / `?` / `??` 后缀直接附加在标识符上，将开发者对代码区域的“信心等级”和“锁定意图”写入源文件，由编译器和 AI 工具严格执行。

- 对比 Rust：属性宏 `#[allow(...)]` 或注释，缺乏结构化的确定性信号。
- 对比契约式语言（如 Dafny, Eiffel）：它们关注逻辑正确性，但不区分“人类已确认”和“AI 可修改”的部分。
- Mimi 通过后缀锁，将 **软件工程的置信度管理** 嵌入语法树。在主流系统语言中，把“人类锁定/AI 可改”的边界作为一等语法构造的做法较为少见。

### 4.2 结构化 Saga 补偿为一等语言构造
`on failure` 块与 `parasteps` 组合，提供原生的分布式事务补偿语义，由编译器自动生成补偿栈的入栈和逆序执行代码。

- 对比：微服务框架中 Saga 模式通常由手动编排或第三方库实现；Rust 中通过 `Drop` 和 RAII 可实现清理，但无法干净表达“仅当事务路径失败时逆序补偿”且补偿本身可能失败的情况。
- Mimi 将业务补偿逻辑提升到语法层面，并和错误处理、并发取消深度集成。这种原生集成在主流系统语言中并不常见。

### 4.3 多态碎片生长：从 `...` 到可执行
整个文件的生命周期都是合法的：从全篇 `desc` 和 `...` 的草图，逐步填实为精确代码，同一文件同一事实源，中间产物永不废弃。

- 对比：UML 或其他建模语言需与代码保持双向同步；Rust 等传统语言没有等效的渐进式草图机制。
- Mimi 的“渐进增强”过程使软件设计不再是一次性抛出的文档，而是代码的**成长年轮**。从模糊意图到精确实现共用同一源文件的工作流，是 Mimi 区别于现有语言的核心体验之一。

### 4.4 契约、数学与补偿的统一嵌入
`requires`/`ensures`/`math:` 块和 `on failure` 共存于同一函数签名的同一作用域，而不仅作为注释或外部验证工具的输入。编译器在特定模式下可验证、可用数学推导形状、可为 AI 提供精确上下文。

- 对比：Ada/SPARK 有契约，但缺少自然语言意图与并发补偿；Swift 有 `async`/`await` 但无 Saga；Python 有 `doctest` 但无结构化补偿。
- Mimi 将契约、数学推导和 Saga 补偿融合在同一作用域内。这种三者的原生组合在主流系统语言中较为罕见，使其更贴近服务端事务编程场景。

### 4.5 线性能力的安全性与可组合性
`cap` 类型强制权限的显式传递，避免隐式环境权限，使 I/O 等敏感操作完全可审计。

- 对比：Rust 没有内建的能力类型；Haskell 的 `IO` monad 将 I/O 隐式地封装在类型中；Idris 等语言有线性类型但复杂度较高。
- Mimi 用 `cap` 提供了一种轻量级、组合式的能力系统。在面向实用的系统语言里，把线性能力作为一等类型并和文件/网络权限直接结合的做法，属于少见的差异化设计。

### 4.6 Mimi 的精确切入点：面向 AI 协作的意图锁定语言
Mimi 不是在单纯追求“比 Rust 更安全”或“比 Python 更快”。它的精确切入点是：

> **把“人类决策与 AI 生成的边界”作为语言的一等公民，让编译器和 IDE 强制执行这条边界，从而支持可信的人机协同软件构造。**

现有语言默认代码完全由人类编写，AI 工具只是外部辅助；而 Mimi 从语法层面为 AI 协作设计了基础设施：

- `?` / `??` 标记 AI 可生成、可修改的区域；
- `$` / `$$` 标记人类已锁定、AI 不得触碰的区域；
- `desc` / `rule` / `requires` / `ensures` 提供机器可读的意图上下文；
- `parasteps` / `on failure` 为 AI 生成的并发/事务代码提供安全骨架；
- `cap` 让 AI 生成的代码无法隐式获得 I/O 权限。

因此，Mimi 的独特价值主张可以概括为 **“Intent-as-Code + Safe AI Collaboration”**：它是为“人类与 AI 共同编写生产代码”这一场景设计的系统语言。它不与 Rust 比拼裸性能，也不与 Python 比拼原型速度，而是在 **“AI 辅助软件构造的可信度”** 这一维度上建立差异。

当然，这一差异化能否成为真正的市场切入点，取决于实际体验：锁后缀是否足够轻量、AI 工具是否真正尊重锁、以及 `.mms → .mimi → 可执行代码` 的链路是否足够顺畅。

## 五、特性裁剪：保留与推迟/移除

| 特性 | 决策 | 理由 |
|------|------|------|
| **Move / 借用 / 生命周期** | **保留**，生命周期基本推断，极少量显式注解 | 内存安全基础，已有 Rust 降低的门槛 |
| **`shared` / `local_shared` / `weak`** | **保留** | 简洁统多所有权，比 Rust 的 `Arc<RwLock<T>>` 直观 |
| **Arena + `ref`** | **保留** | 大幅简化区域内存使用 |
| **`cap` 线性能力** | **保留**，但暂时不引入组合分配语法，仅支持单个 `cap` 的传递 | 组合能力（`+`）保留语法，但标准库仅提供基本 `cap`，组合的实现可以后续优化 |
| **`actor`** | **保留** | 并发模型核心，门槛远低于手动 `tokio`/锁 |
| **`parasteps` + `on failure`** | **保留** | 语言最大亮点，高价值 |
| **`requires / ensures / math:`** | **保留**，但`--verify-contracts` 模式推迟到 v1.1 | v1.0 仅支持静态简单检查，保证可用 |
| **`desc` 和 `rule` 编译期审计** | `rule` 审计推迟；`desc` 作为元数据保留 | 依赖 AI 的审计暂时不可靠，不作为硬性编译门槛 |
| **`comptime` 元编程** | **裁剪为最小可用**：仅支持 `quote!` 生成代码，暂不支持 `type_info` 反射 | 避免实现复杂度爆炸，v1.0 只提供声明宏级别的代码生成 |
| **`interface` / `trait`** | **移除**，改用 duck typing + `comptime` 静态分发 | trait 系统是 Rust 学习曲线的陡峭部分，移除显著降低门槛；泛型通过 `comptime` 静态分派或简单的 `where T: ...` 未来扩展 |
| **`match` 穷尽性 / ADT** | **保留** | 现代语言标配，门槛低 |
| **模式匹配与解构** | **保留** | 基本语法，必需 |
| **`drop` 显式丢弃** | **保留** | 与所有权系统一致 |
| **`newtype`** | **新增**，解决别名歧义 | 轻量，无学习负担 |
| **`use`** | **Mimi 导入语法** | `@import` 属于 MimiSpec |

**裁剪后核心特性集**：所有权/借用、ADT+模式匹配、actor、parasteps/补偿、契约基础、cap 线性传递、Arena、意图后缀。这套组合在主流系统语言中具有较强的差异化，同时整体规格仍比 Rust 更紧凑。

## 六、对比 Rust 的门槛分析

| Rust 痛点 | Mimi 对策 |
|-----------|----------|
| 生命周期标注泛滥 | 编译器自动推断 >95% 生命周期，仅复杂场景需显式 `'a`（语法与 Rust 相同，但非必须） |
| `Rc` / `Arc` / `RefCell` 组合繁琐 | 提供 `shared T`（内部可变性由编译器自动插入 `Mutex` 或单线程检查） |
| trait 与泛型约束系统复杂 | v1.0 无 trait，用 duck typing，降低早期学习压力 |
| 异步编程复杂（tokio, Pin） | 内建 actor 和 `spawn`/`await`，无底层运行时接触 |
| 错误处理链长 | `?` 传播 + `on failure` 补偿清晰 |
| 宏系统卫生性难懂 | 仅提供类声明宏 `quote!`，简单直观 |
| 所有权陡峭曲线 | 提供“默认 Copy”规则和 Arena 作为逃生口，新手先按直觉写，编译器友好纠错 |

**结论**：Mimi v1.0 的门槛显著低于 Rust，但在内存安全和并发正确性上仍提供等价保障。
