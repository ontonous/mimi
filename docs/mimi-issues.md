# Mimi 语言当前问题清单

> 审计日期: 2026-06-24 (updated v0.22 progress)
> 审计背景: v0.22 尝试用 Mimi 实现 mimispec 解析器（7,290 行 Rust → Mimi）过程中发现

## 一、类型系统缺陷

### 1.1 ~~缺少 `Option<T>`~~ ✅ v0.22 已修复

**症状**: 无法表达"可能有值也可能没有"的语义。
**影响**: ⚫ 严重。每个可选字段都需要 `has_xxx: bool` + 默认值 workaround，污染所有 API。
**位置**: `src/core/ast.rs` 有 `Type::Option(Box<Type>)` 类型。
**修复**: v0.22 补齐 `Option<T>` 的类型推断、codegen、interpreter 全路径，L1 双后端测试通过。

### 1.2 `Result<T, E>` 不完整

**症状**: 
- `Result` 在 type checker 中已存在，但 codegen/interp 路径不一致
- `?` 运算符存在但只支持特定的错误类型
**影响**: ⚪ 中等。`Result` 可用于简单场景，但复杂嵌套泛型场景有 bug。

### 1.3 ~~不支持递归类型~~ ✅ v0.22 已修复

**症状**: 无法定义 `type Expr { Call(name: string, args: List<Expr>) }`。
**影响**: ⚫ 严重。任何树形结构（AST、JSON）都无法用原生类型表达。
**原因**: 类型检查器在 `check_type` 时对自引用类型不做特殊处理。
**修复**: v0.22 在类型检查器中支持 Record/Union/Enum 自引用类型定义，L1 测试通过。

### 1.4 泛型 `List<T>` 嵌套（codegen 有限制）

**症状**: `List<List<i32>>` 类型标注通过，codegen 嵌套索引仅支持 i64 元素。
**影响**: ⚪ 中等。`List<List<Atom>>` 嵌套在解释器中可用，codegen 需扩展。
**注意**: v0.22 完成类型推断 + 解释器支持，codegen 列表元素 type-erased 为 i64。

### 1.5 不支持函数类型作为参数

**症状**: `func(List<T>, i32) -> (Any, i32)` 在高阶函数中解析失败。
**影响**: ⚪ 中等。Paser 的 block 解析需要回调函数模式。

---

## 二、字符串/字符处理不足

### 2.1 没有字符类型

**症状**: 所有字符操作通过 `char_at(s, i)` 返回单字符 string。
**影响**: ⚪ 中等。性能差 + API 别扭，但不完全阻塞。
**需要的 API**: `char_code(s, i) -> i32`, `chr(i) -> string`。

### 2.2 缺少正则表达式匹配

**症状**: 字符分类需要手写 `c >= "a" && c <= "z"`。
**影响**: 🟢 低。手写没问题但冗长。

### 2.3 `str_starts_with` 不是字符级操作

**症状**: 无法高效地逐个字符遍历字符串并做模式匹配。
**影响**: 🟢 低。可以用 `char_at` + 索引循环替代。

---

## 三、语法/Parser 限制

### 3.1 缩进敏感 + 没有行延续

**症状**: 长表达式不能用 `||` 续行，必须写在一行。
**影响**: ⚪ 中等。`is_keyword_ident` 这种函数需要 30 个 `||` 挤在一行。
**示例**:
```mimi
// 在 Mimi 中必须写成一整行:
func is_keyword_ident(k: string) -> bool {
    k == "a" || k == "b" || k == "c" || ...(30个)...
}
// 不能写成:
func is_keyword_ident(k: string) -> bool {
    k == "a" || k == "b" || k == "c"
    || k == "d"   // 语法错误
}
```

### 3.2 不支持 `use module::function` 导入单个函数

**症状**: 必须 `use module` 然后 `module::function()`。
**影响**: 🟢 低。只是语法糖缺失，不影响功能。

### 3.3 `else if` 解析问题

**症状**: `if ... { } else if ... { }` 在某些嵌套场景下解析失败。
**影响**: 🟢 低。大部分场景可用。

---

## 四、标准库不完整

### 4.1 缺少集合类型

| 缺失类型 | 影响 | 替代方案 |
|----------|------|----------|
| `Map<K, V>` 不可在 Mimi 源码中创建 | 🟢 低 | `from_list` 内置函数 |
| `Set<T>` | 🟢 低 | 用 Map 模拟 |
| 迭代器 | 🟢 低 | `for i in range` |

### 4.2 缺少 JSON 序列化/反序列化

- `to_json` / `from_json` 已存在
- 但 `Record` 类型的字段访问无法在 Mimi 中动态进行
- `json_get_string(obj, key)` 需要传 JSON 字符串而不是 Record

### 4.3 缺少正则表达式

- 存在 `regex_match` / `regex_find` / `regex_replace` 内置函数
- 但在 Mimi 中作为内置函数可用，不需要额外 stdlib

---

## 五、Codegen/运行时问题

### 5.1 LLVM codegen 对复杂类型的支持不稳定

- `List<List<T>>` 嵌套在 codegen 路径可能崩溃
- 某些高级类型组合在 LLVM IR 生成时产生无效 IR

### 5.2 解释器与 codegen 行为不完全一致

- 某些边缘情况（复杂泛型、嵌套调用）双后端行为不同
- 已知 11 个 `dual_gap_*` 已在 v0.21 关闭，但可能还有未发现的差距

---

## 六、开发者体验

### 6.1 错误消息不够友好

- 某些解析错误缺乏源码位置信息
- 缺少 "did you mean?" 建议

### 6.2 缺少字符串插值

- 没有 `"hello {name}"` 语法
- 只能用 `"hello " + name` 拼接

---

| 问题 | 严重性 | 对自举的阻塞程度 | 修复难度 |
|------|--------|-----------------|---------|
| ~~缺少 `Option<T>`~~ (✅ v0.22) | ⚫→✅ | **已解决** | 中 |
| ~~不支持递归类型~~ (✅ v0.22) | ⚫→✅ | **已解决** | 高 |

| 无行延续 | 🟢 | 不阻塞 | 低 |

## 结论

自举（用 Mimi 写 mimispec 解析器）v0.22 后**可行性显著提升**：

1. ~~**缺少 `Option<T>`**~~ — ✅ v0.22 已补齐全路径
2. ~~**不支持递归类型**~~ — ✅ v0.22 类型检查器支持自引用
3. **泛型嵌套** — 🟡 类型推断通过，codegen 列表元素限 i64，List<List<Atom>> 在解释器中可用
4. ~~**字符串能力**~~ — ✅ `chr`/`char_code` 补齐

剩余限制：
- Codegen 列表元素仅支持标量类型（`List<List<T>>` 嵌套索引在 codegen 中不可用）
- 复杂高阶函数签名待完善

v0.22 推进后，mimispec 解析器的核心阻塞点（Option<T>、递归类型）已清除。
